一. HTTP报文

　　HTTP报文是在HTTP应用程序之间发送的数据块.报文由三个部分组成: 起始行(start line) 头部(header) 主体(body);
　　所有HTTP报文都可以分为两类,请求报文(request message)和响应报文(response message);
　　格式如下:

1.报文格式

  • 方法(method): 客户端请求的方法, 入 GET POST 等;
  • 请求URL(request-url): 请求的资源在服务器上的绝对路径;
  • 协议版本(version): 格式为HTTP/<major>.<minor> (主要版本号.次要版本号),如 HTTP/1.1;
  • 头部信息(headers): 包含一些附加信息,头部信息有很多种,但不是每一个都需要在客户端与服务端交互中传递;
  • 主体(body): 包含了一些数据.

2.报文详解(起始行—header—body)

2.1 起始行
  • 请求行
    GET /data/getModelLists.json HTTP/1.1
    这里包含了三部分信息: 请求方法(GET), URL(/data/getModelLists.json),协议版本(HTTP/1.1);
  • 响应行
    HTTP/1.1 200 OK
    协议版本 HTTP/1.1, 状态码: 200, 原因短语: OK

请求方法: 常见的有GET,POST,还有DELETE, PUT等等,
HTTP设计的易于拓展,所以可以根据规范实现自己的一套方法.

状态码: 状态码(status code)是服务器完成请求后返回给服务器的一个标识,告诉客户端发生了什么事,因为请求并不一定是成功的.
返回的状态码方便程序进行错误处理,而原因短语便于人们理解.

2.2 首部

　　首部向请求报文和响应报文中添加了一些附加信息.本质上来说,是一些键值对;
　　http/1.1 首部规范和描述

HTTP规范定义了集中首部字段,HTTP首部通常可以分为以下几类:

  • 通用首部: 可在请求报文中出现, 也可在响应报文中出现;
    　　比如: Date:Mon, 28 Aug 2017 02:08:04 GMT 构建报文的时间和日期;
  • 请求首部: 提供更多有关请求的信息;
    　　比如: Accept: */*,image/*,text/html 客户端向服务端表明能接受的资源类型,分别为任意类型,所有图片类型和html;
  • 响应首部: 提供更多有关响应的信息;
    　　比如: Server: Tengine 告知客户端它在与Tengine服务器进行交互;
  • 实体首部: 描述主体的长度和内容,或者资源自身;
    　　比如: Content-Type: text/html;charset=utf-8 表明主体数据是以utf-8字符集表示的html文档;
  • 扩展首部: 规范中没有定义的新首部;
    　　非标准首部,由开发者创建; 　　

    二. HTTP缓存

    2.1 关于浏览器缓存

    　　客户端通过HTTP请求向服务端请求资源,但是HTTP请求本身是要消耗资源的,对于同一个资源如果每次都向服务端发起请求,势必产生不必要的消耗,也会加大服务器的负载.如果对于那些尚未变化的资源能够从缓存中获取,不必在向服务端请求,则方便许多;
    　　使用缓存有以下优点:
  • 减少了冗余数据的传输; 节省了宽带费用
  • 缓解了网络瓶颈的问题; 更快的加载页面
  • 降低了对原始服务器的要求; 避免服务器过载
  • 降低了距离延时;

　　为了更好地了解浏览器的缓存机制,可以将HTTP缓存体系分为三大部分:

  • 缓存存储策略
  • 缓存过期策略
  • 缓存对比策略

2.1.1 缓存存储策略(Cache-Control)

　　用来确定HTTP相应内容是否允许被缓存,以及被哪些客户端缓存.对应请求头中的Cache-Control设置.(http/1.1)
　　首先看看Cache-Control的可选值有哪些:
　　 Public 　　 Private
　　 no-cache 　　 max-age
　　* no-store
　　前4个都会缓存文件数据（关于 no-cache 应理解为“不建议使用本地缓存”，其仍然会缓存数据到本地），后者 no-store 则不会在客户端缓存任何响应数据.
　　即便我们设置了Cache-Control,比如为Public,也不意味着浏览器会直接从缓存中读取数据并使用,因为浏览器并不能确定本地缓存的数据是否可用(有可能已经过期失效),所以还需要通过缓存过期机制来确认.

2.1.2 缓存过期策略(Expires)

　　客户端用来确认本地缓存的数据是否过期,来决定是否直接从本地缓存数据中加载数据并展示（否则就发请求到服务端获取）.
　　这里就用到了Expires,Expires是服务端返回的一个绝对时间,例如: Expires: Thu, 01 Dec 1994 16:00:00 GMT.客户端在比照本地时间点后,如果在过期时间内,则直接从缓存中加载,如果超过了过期时间,则说明本地缓存过期,需要重新向服务端发送请求.
　　需要注意的是,如果在Cache-Control中设置了过期策略,其优先级会高于Expires,比如Cache-Control=max-age(no-cache);这两则既包含了缓存存储策略也包含了过期策略;
　　同时,Expires 的一个缺点就是，返回的到期时间是服务器端的时间，这样存在一个问题，如果浏览器所在机器的时间与服务器的时间相差很大，那么误差就很大，所以在HTTP 1.1版开始，使用Cache-Control: max-age替代. 如果max-age和Expires同时存在，则被Cache-Control的max-age覆盖。

而no-cache则相当于上面的maxAge = 0;

2.1.3 缓存对比策略

　　在客户端重新想服务端发送请求后,服务端通过比对标识,来判断客户端的缓存数据是否还有效,如果对比标识后有效,则返回304告诉客户端从缓存中获取数据,否则服务端会重新发送新的数据.
　　服务器需要客户端提供的对比标识( If-Modified-Since、If-None-Match ),这里要注意的是,服务端要在首次响应式向客户端传递( Last-Modified、ETags )到客户端

2.2 浏览器缓存过程

  • 在客户端首次向服务端发起请求时,请求头中带有Cache-Control,分别可能有Public,Private,max-age,no-cache,no-store几种情况;
  • 服务端响应请求,如果请求成功,则返回数据,同时响应头中会带有Expires或者last-modified或Etag(被请求变量的实体标记)的标识;
  • 客户端再次请求资源,
    • 如果Cache-Control=Public/Private,并且检查Expires后,如果本地缓存没有过期,则直接从本地获取数据,如果已经过期,则重新向服务端发送请求;
    • 如果Cache-Control=max-age,则Expires：当前客户端时间 + maxAge,同上;
    • 如果Cache-Control=no-cache,则本地缓存立即失效,重新向服务端发送请求;
      如果重新向服务端发送请求,则根据HTTP协议规定,浏览器会向服务器传送If-Modified-Since或If-None-Match报头(Request Header)，询问该时间之后文件是否有被修改过;
  • 服务端接收到请求后,会通过If-Modified-Since或者If-None-Match的标识,判断服务端资源是否修改过,如果对比成功,则返回304和空的body,告诉客户端直接从缓存中拿取数据;如果对比失败,则重新发送新的数据给客户端.
